           WHY: mounts the Mini CMS under /admin as required by A03.

           WHY: server only starts listening once the database is reachable.
                If the connection fails the process exits with an explanation
                rather than starting a broken server.


── MONGOOSE MODELS (all new) ───────────────────────────────────────────────────

models/Category.js  (new file)
  WHY: A03 requires a Category collection. Centralizing the schema in its own
       file keeps model definitions separate from route logic. Fields:
       - name (String, required): human-readable label
       - slug (String, required, unique): URL-safe identifier used in routes
         like /projects/category/:slug
       - description (String, optional): shown in the admin list

models/Project.js  (new file)
  WHY: replaces the JSON array in data/projects.json. Mongoose provides
       schema enforcement and query helpers. Fields:
       - slug (String, required, unique): used for /projects/:slug routes
       - title, description (String, required): core display fields
       - isActive (Boolean, required): controls public visibility; only
         isActive === true projects appear on the public site and API
       - tags ([{ name: String }]): embedded sub-documents instead of plain
         strings so they can be queried with dot-notation ("tags.name")
       - categoryId (ObjectId, ref Category, required): foreign key link;
         using .populate("categoryId") in queries joins in the full category
         object

models/Contact.js  (new file)
  WHY: A03 requires contact form submissions to be saved to MongoDB.
       Previously POST /contact only rendered a response without persisting
       anything. Fields:
       - name, email, message (String, required): form inputs
       - postedDate (Date, default Date.now): timestamp for sorting in admin
       - isRead (Boolean, default false): lets admin track new submissions


── SERVICES (new) ──────────────────────────────────────────────────────────────

services/projectService.js  (new file)
  WHY: the requirements explicitly state "centralized helper/service for
       filtering logic — no copy-paste query logic across routes". Without
       this, both pageRoutes and apiRoutes would duplicate the same Mongoose
       filter-building logic for ?q= and ?tag= queries.

  buildFilter(q, tag, extraFilter)
       Builds a Mongoose filter object from optional search term, tag, and
       a base filter (e.g. { isActive: true }). Uses $or with regex for the
       text search and a case-insensitive regex for tag matching. Used by all
       three query entry points.

  getActiveProjects(q, tag)
       Calls buildFilter with isActive: true, then runs Project.find().
       Used by GET /projects and GET /api/projects.

  getActiveProjectsByCategorySlug(slug, q, tag)
       Looks up the category by slug first, then filters projects by
       categoryId. Returns both the category and the projects so the view
       can display the category name. Used by GET /projects/category/:slug
       and GET /api/projects/category/:slug.


── ROUTES (all new files) ──────────────────────────────────────────────────────

routes/pageRoutes.js  (new file)
  Replaces: routers/pageRouter.js (old A01 file that used res.sendFile)
  WHY: A02's pageRouter.js still called res.sendFile() pointing at static
       .html files in a /pages/ directory — it was never updated to use EJS.
       The new file uses res.render() and pulls live data from MongoDB.

  GET /
       Renders index.ejs. No data needed.

  GET /about
       Renders about.ejs. No data needed.

  GET /projects  (supports ?q= and ?tag=)
       Calls getActiveProjects() from projectService.
       Passes q and tag to the view so the search form can repopulate and
       display filter info.

  GET /projects/category/:slug
       Calls getActiveProjectsByCategorySlug().
       Returns 404 if the category slug is not found.
       Passes category to projects.ejs so the heading updates to
       "Projects in <category name>".
       NOTE: this route is defined BEFORE /projects/:slug to prevent Express
       from interpreting "category" as a slug value.

  GET /projects/:slug
       Finds one active project by slug with .populate("categoryId").
       Also loads other active projects for the sidebar.
       Returns 404 if the slug is not found or the project is inactive.

  GET /contact
       Renders contact.ejs with empty success/error/fields state.

  POST /contact
       Validates that name, email, and message are all present.
       On missing fields: re-renders with an error and repopulates the form.
       On success: saves a Contact document and re-renders with a success
       message (same view, no redirect needed).

routes/apiRoutes.js  (new file)
  Replaces: routers/apiRouter.js (old file reading from projects.json)
  WHY: old router read from a static JSON file; A03 requires all data to
       come from MongoDB.

  GET /api/projects  (supports ?q= and ?tag=)
       Returns active projects as JSON using getActiveProjects().

  GET /api/projects/category/:slug
       NOTE: defined before /api/projects/:id so "category" is not treated
       as a MongoDB ObjectId.
       Returns active projects for a category as JSON, or 404 if the
       category slug doesn't exist.

  GET /api/projects/:id
       Finds a project by MongoDB _id.
       Handles CastError (invalid ObjectId format) gracefully as a 404.

  GET /api/categories
       Returns all categories as JSON (read-only per requirements).

routes/adminRoutes.js  (new file)
  WHY: A03 requires a Mini CMS for managing categories, projects, and
       contact submissions. No equivalent existed in A02.

  GET /admin
       Renders the dashboard with links to each admin section.

  -- Contacts --
  GET /admin/contacts
       Lists all submissions sorted newest-first.

  PATCH /admin/contacts/:id/read
       Toggles the isRead boolean and returns JSON with the new state.
       The view uses fetch() to update the UI without a page reload.

  DELETE /admin/contacts/:id
       Deletes the document and returns JSON. The view removes the row
       from the DOM on success.

  -- Categories --
  GET /admin/categories
       Lists categories with a per-category project count. The count is
       computed via Project.aggregate() and merged into each category object
       so the view can disable the Delete button when count > 0.

  GET /admin/categories/new
  POST /admin/categories
       Create flow. On duplicate slug (code 11000) re-renders the form with
       a friendly message instead of crashing.

  GET /admin/categories/:id/edit
  POST /admin/categories/:id
       Update flow. Same duplicate-slug handling.

  DELETE /admin/categories/:id
       SAFE DELETION: counts projects referencing the category first. If
       count > 0 returns HTTP 400 with a friendly error message. The UI also
       disables the button when count > 0, but the server enforces it
       independently. Returns JSON.

  -- Projects --
  GET /admin/projects
       Lists all projects (active and inactive) with category and tags.

  GET /admin/projects/new
  POST /admin/projects
       Create flow. Tags are entered as a comma-separated string in the form
       and converted to [{ name }] objects before saving. isActive comes in
       as the string "true"/"false" from the form and is coerced to Boolean.

  GET /admin/projects/:id/edit
       Loads the project and converts the tags array back to a
       comma-separated string (tagsString) for the form input.

  POST /admin/projects/:id
       Update flow. Same tag and isActive handling as create.

  DELETE /admin/projects/:id
       Deletes and returns JSON.
